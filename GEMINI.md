
# Repository Overview

This repo contains the following services and utilities all written in the Go programming language:

* `app-exposer` - Accepts JSON encoded analysis descriptions from upstream services, turns them into JSON encoded analysis bundles that contain K8s-resource definitions. Those are sent to a vice-operator.

* `vice-operator` - Can run in a different Kubernetes cluster than the rest of the Discovery Environment. Accepts JSON encoded resource bundles from app-exposer, performs cluster-specific transformations on the bundles, and submits the resulting resource definitions to the Kubernetes API. vice-operator is stateless and does not connect to other Discovery Environment services. Provides an HTTP+JSON API protected by basic auth and a Swagger UI for interacting with the API.

* `vice-list` - A command-line utility for listing available VICE apps from the DE database. Intended for development purposes.

* `vice-export` - A command-line utility for exporting a VICE app to a JSON file. Intended for development purposes.

* `vice-import` - A command-line utility for importing a VICE app from a JSON file. Intended for development purposes.

* `vice-launch` - A command-line utility for launching a VICE analysis from a JSON file generated by `vice-export`. Intended for development purposes.

* `vice-userid` - A command-line utility for looking up a user's user ID. Intended for development purposes.

* `vice-bundle` - A command-line utility that generates an analysis bundle that can be sent to vice-operator. Intended for development purposes.

* `workflow-builder` - A command-line utility to submit a batch analysis. Intended for development purposes.

# Building the project

Most build related operations are orchestrated using the `just` command-line tool and the Justfile included in the Git repository.

To build everything, run `just build`.

To clean the local box of builds, run `just clean`.

To run tests, run `just test`.

To build a specific service or tool, run `just <service-or-tool-name>`. For example, `just app-exposer` builds `app-exposer`.

Since the projects in the app-exposer Git repository can share code, it's usually a good idea to build everything occasionally during development just to make sure something obvious isn't broken.

# Go Coding Standards & Architecture

## General Style
* **Succinctness**: Prefer concise, idiomatic Go. Avoid unnecessary verbosity or overly complex type hierarchies.
* **File Structure**: Split files exceeding ~300 lines by entity or domain type (e.g., one file per database entity).
* **Composition**: Favor composition (embedding and interfaces) over deep inheritance hierarchies.
* **Standard Library**: Always prefer the standard library over adding new dependencies unless the dependency is the industry standard (e.g., `echo`, `logrus`).
* **Interfaces**: Define interfaces near their implementation, not in the consumer package. Before defining a new interface, check for an existing one with the same method set.

## Error Handling
* **Never ignore errors**: Every returned error must be handled or explicitly discarded with a justifying comment.
* **Typed Errors**: Use custom error types (retrieved via `errors.As`) for domain conditions like "not found". Do not use string matching on error messages.
* **Database Errors**: Always check `sql.ErrNoRows` separately from other database failures; a missing row is not a system error.

## Context & Database Patterns
* **Context Threading**: Always thread `context.Context` through the full call chain.
  - In Echo handlers, extract context with `c.Request().Context()`.
  - Use `*Context` variants for all DB operations (e.g., `QueryRowContext`, `ExecContext`).
* **Transaction Discipline**: All database functions must accept a transaction (`Tx`) parameter. Functions should not operate outside of a transaction unless they are the ones initiating the transaction.
* **Dependency Injection**: Use constructor injection for dependencies like DB connections, loggers, and HTTP clients. Avoid package-level variables for stateful objects.

## API Design
* **Idempotency**: DELETE operations must be idempotent. Deleting a resource that does not exist should return success, not a "not found" error.
* **API Contracts**: Match original API contracts exactly when porting services. Avoid adding optionality to update requests unless supported by the original API.

## Testing
* **Table-Driven Tests**: Prefer table-driven tests for comprehensive coverage of multiple scenarios in a single test function.
